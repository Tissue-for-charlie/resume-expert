# resume-expert

一份通用的 **AI Agent 操作手册**（System Prompt / Custom Instructions），让任意 AI 助手具备行业自适应的简历生成、深度评审与增量修改能力。不是简单的"填表生成器"——它内置了招聘市场知识（ATS 筛选规则、行业×职级差异化策略、中英文简历规范），让 AI 像一位资深职业顾问一样工作。

### 适配平台

| 平台 | 使用方式 |
|------|----------|
| **Claude Code** | 放入项目目录，自动识别为 Skill |
| **Cursor** | 粘贴到 `.cursorrules` 或 Cursor Settings → Rules |
| **GitHub Copilot** | 粘贴到 `.github/copilot-instructions.md` |
| **ChatGPT Custom GPT** | 粘贴到 GPT 配置的 Instructions 字段 |
| **Windsurf** | 粘贴到 `.windsurfrules` |
| **任意 LLM 客户端** | 作为 System Prompt 粘贴，配合对话使用 |

核心文件只有一个：[`SKILL.md`](./SKILL.md)——约 440 行的纯文本操作手册，零依赖，直接使用。

## 为什么用这个 Skill？

普通 AI 写简历有三个通病：**规格说明书风格**（"该系统包括 A、B、C 功能"）、**信息堆砌无优先级**（什么都写等于什么都没写）、**不懂 ATS 机器筛选规则**（排版漂亮但过不了系统关）。

这个 Skill 解决的就是这三个问题。

### 核心理念

- **招聘者在一份简历上平均停留 6-10 秒**——信息优先级决定了会不会被细读
- **先过机器再过人**——超过 75% 的简历在 ATS（Applicant Tracking System）关被筛掉，原因往往不是资历不够，而是格式或关键词问题
- **写得少但对，比写得多但废话更有效**——不写自我评价段落，用项目经历证明能力
- **不同行业、不同职级的简历，核心卖点完全不同**——互联网大厂 ≠ 国企 ≠ 外企 ≠ 初创公司

## 三种工作模式

Skill 会根据用户意图自动路由到对应模式，无需手动切换：

### 模式 A：从零生成简历

```
触发：用户说"帮我做一份简历" / "写简历"
输入：用户零散的经历描述
输出：格式规范的 .docx 文件
```

完整流程 7 步：

1. **确定方向** — 目标行业、语言、页数。如果用户只提供了姓名+电话+目标岗位，先出"骨架简历"让用户看到进度，缺失项标注 `[待补充]`，后续逐段填充
2. **分维度收集信息** — 8 个维度（基本信息→教育→求职意向→项目→工作→竞赛→作品→到岗），每次聚焦 1 个维度，已提供的信息自动跳过。支持批量模式：当用户想快速推进时，合并所有剩余维度一次性询问
3. **设计结构** — 根据行业×职级矩阵确定模块顺序和篇幅分配。技术岗项目经历占 60-70%，非技术岗工作经历放前面
4. **项目描述重写** — 这是核心步骤。将"规格说明书风格"或"任务清单风格"的原始描述重写为个人实践叙事：主动动词开头、技术名词融入上下文、补充量化结果
5. **生成 DOCX** — 用 docx-js（Node.js）生成，自动设置字体/字号/颜色/章节分隔线/bullet 编号/可点击超链接
6. **验证** — python-docx 逐段核对文本 + 检查 font 非空 + 确认超链接存在 + 篇幅检查
7. **失败回退** — 如果 docx-js 不可用，自动降级到 python-docx → Markdown 版本，每层都告知用户

### 模式 B：评审已有简历

```
触发：用户说"帮我看一下这份简历" / "这个简历行不行" / 附带 .docx 文件
输入：用户上传的简历文件（docx/pdf）或粘贴的简历原文
输出：100 分制评分 + 四列模板的具体修改建议
```

1. **深度提取** — python-docx 提取全文文本 + 排版参数（字体/null 检查、字号层级、超链接）。文件损坏时给 3 个选项（重新上传/粘贴原文/切换到模式 A），不自动重试
2. **四维度评分**（可根据行业调整权重）：

| 维度 | 默认权重 | 考察什么 |
|------|----------|----------|
| 内容质量 | 40% | 项目描述是否有结构、有量化数据 |
| 排版可读性 | 25% | 字号层级、info dump 检测 |
| 关键词覆盖 | 20% | 目标行业核心关键词命中率 |
| 差异化定位 | 15% | 面试官能否找到"想聊一下"的钩子 |

3. **分级反馈** — 每条反馈遵循四列模板（原文 → 问题 → 建议替换 → 替换理由），可直接替换的具体文案，不写"建议优化措辞"这种空话
4. **5 级评分锚点** — 从 <40（建议重写）到 90-100（行业标杆），保证不同对话中评分一致
5. **自动过渡到修改模式** — 评审完主动问"需要我逐条帮你改吗？"

### 模式 C：增量修改已有简历

```
触发：用户说"帮我把这段改一下" / "换个说法" / "加个项目"
输入：定位到具体段落/章节的修改指令
输出：修改后的 .docx 或纯文本替换建议
```

1. **精准定位** — 支持段落级、章节级、全局搜索替换、模糊意图（先诊断再修改）
2. **策略选择** — 新增用模式 A 的重写标准、优化复用对照表、删除先确认、结构调整参考行业矩阵
3. **输出方式** — DOCX 修改优先保留原格式（python-docx → docx-js 重建 → 纯文本，三层回退）
4. **四项修改后一致性检查**：
   - 求职意向是否仍匹配新增内容
   - 技术栈在项目描述和技能区间是否一致
   - 时态是否统一
   - Bullet 密度是否平衡
5. **修改记录追踪** — 同份简历累积 ≥3 次修改后，自动记录演变历史

## 行业 × 职级决策矩阵

这是 Skill 的"北极星"——确定用户目标行业和职级后，所有内容决策都围绕对应的单元格展开：

| 行业 | 实习/低年级 | 校招/应届 | 社招/跳槽 |
|------|------------|----------|----------|
| **互联网/科技** | 项目经历碾压；GitHub 是关键证据 | 基础扎实 + 深度项目；竞赛加分 | 业务影响力和量化产出 |
| **金融/银行** | GPA + 学校 + 相关比赛 | 证书 + 实习 + 行业认知 | 交易额/项目规模/合规意识 |
| **外企/咨询** | 国际经历、英语、课外领导力 | 咨询实习 + 案例能力 | 项目规模、跨文化协作 |
| **国企/体制内** | 学校层次 + 专业对口 + 学生干部 | 学历（硕士优先）+ 证书 | 职称/资质、项目体量、稳定性 |
| **产品/运营** | 校内实践、社团、自建小产品 | 数据思维、从 0 到 1 案例 | 业务指标（DAU/留存/GMV） |
| **设计/创意** | 作品集链接放最前面 | 实习 + 工具链 + 设计方法论 | 设计系统构建能力 |

## ATS 兼容性指南（内置于 Skill 指令中）

现代招聘流程中，简历先过 ATS 再过人。Skill 内建了完整的 ATS 避坑规则：

### 格式避坑
- ❌ 表格/文本框、图片中的文字、双栏布局、自定义字体 → 全部禁止
- ✅ 线性流式布局、纯文本呈现、系统通用字体、DOCX 格式优先

### 关键词策略
- 从目标岗位 JD 反推高频术语，确保自然出现在简历中
- 专业术语全称+缩写双写（如"分布式事务（Distributed Transaction）"）
- 技能按领域分组，比散装关键词更易被 ATS 正确归类
- 禁止白色字体藏关键词（作弊行为，多数公司明文禁止）

### DOCX vs PDF 投递策略
| 场景 | 推荐 | 原因 |
|------|------|------|
| 官网/招聘系统上传 | DOCX | ATS 解析精度最高 |
| 邮件直投/内推 | PDF | 排版稳定不变形 |
| LinkedIn/猎头平台 | PDF | 平台预览兼容 |

## 关键设计决策

### Phase 0：先扫描再说话

每次对话开始时，必须先并行完成 4 项信息预扫描，**不问用户任何问题**：
1. 读取 Memory 系统（技术栈、GitHub ID、过往项目）
2. 搜索本地文件（简历、项目文档）
3. 拉取 GitHub 仓库列表和 README
4. 扫描本地 Skill 项目和依赖清单作为技能佐证

汇总为「已知信息清单」向用户确认。三项来源全空时跳过，不卡在空状态反复询问。

**搜索截断**：单次搜索超过 15 个文件时只取最近 10 个，其余标注"及其他 N 个文件"供确认。

**粘贴检测**：用户首次消息直接粘贴 ≥100 字简历原文 → 视为"本地文件等价内容"，跳过文件搜索。

### 信息源冲突仲裁（约束规则第 0 条）

当 Memory / 本地文件 / GitHub / 用户口述四个来源对同一事实给出不同值时：

```
用户最新口述 > 本地文件（旧简历等） > GitHub 公开数据 > Memory
```

冲突项以 ⚠️ 标注并请求用户确认。

### 模式切换状态机

覆盖 7 条转换路径，用户在任意时刻切换模式都不会卡在中间状态：
- A 中途→B / A 中途→C / A 生成后→B
- B 完成→C / C 进行中→B / C 进行中→A
- 任意模式→放弃

### 三层回退链

所有涉及文件生成的操作都有明确的降级路径：
- docx-js 失败 → 诊断修复 → python-docx → Markdown（模式 A）
- python-docx 失败 → 给用户三个选项而非自动重试（模式 B）
- python-docx 失败 → docx-js 重建 → 纯文本（模式 C）

### 隐私保护（跨模式强制生效）

以下信息**绝对不写入简历**：身份证号/护照号、完整家庭住址（仅到城市/区）、具体出生日期（仅年份或年龄）、银行账户/薪资期望、婚姻/宗教/政治面貌（国企除外）、照片（日韩等地区除外）。Phase 0 扫描到会主动过滤并说明原因。

### docx-js 正确 API（踩坑总结）

- 列表只用 `numbering config` + `LevelFormat.BULLET`，禁止 unicode 符号模拟
- 所有 TextRun 必须显式设 `font: { ascii, eastAsia }`，不可留 null
- 超链接用 `ExternalHyperlink` 包裹，不要纯文本 URL
- 文件命名 `{姓名}_简历.docx`

## 快速开始

### 1. 使用方式

**直接使用**——把 `SKILL.md` 的内容作为 System Prompt 粘贴到任意 AI 对话中，然后正常对话即可触发：

```
你是一位行业自适应简历策略顾问。以下是你的操作手册：
[粘贴 SKILL.md 全部内容]

---
用户：帮我做一份简历，投字节跳动后端实习
```

**Claude Code 用户**——将本文件夹复制到项目目录，Skill 自动加载。

**Cursor / Copilot 用户**——将 `SKILL.md` 的内容追加到项目的 Rules 文件中（`.cursorrules` 或 `.github/copilot-instructions.md`）。

### 2. 可选依赖（仅生成 docx 时需要）

```bash
npm install docx
```

### 3. 触发关键词

在对话中提及任意简历相关关键词：

| 中文触发词 | 英文触发词 | 路由到 |
|-----------|-----------|--------|
| 制作简历、写简历、生成简历 | — | 模式 A |
| 改简历、优化简历、简历打分、简历评审 | review my resume, fix my resume | 模式 B |
| 帮我看看这份简历 | — | 模式 B |
| 帮我把这段经历改一下 | — | 模式 C |
| 投实习、投秋招、春招简历、社招简历、跳槽 | — | 模式 A（含行业推断） |
| 附带 .docx 或 .pdf 文件 + "帮我看一下" | — | 模式 B |

### 4. 示例对话

```
👤 用户：帮我做一份简历，投字节跳动后端开发实习

🤖 Claude：[Phase 0 自动扫描 Memory + 本地文件]
📋 预扫描结果：
- 从 Memory 获知：Node.js, PHP, MySQL, 网络安全专业, Tissue-for-charlie
- 找到本地文件：项目经历简历描述.md (2024-06-20)
以上信息是否完整准确？可以补充或跳过。

👤 用户：准确

🤖 Claude：[Step 1] 已确认：互联网/科技行业，后端开发实习，中文简历，一页。
[Step 2] 我们先确认基本信息：
- 姓名：杨子菁
- 电话/邮箱：？

👤 用户：15919331052，3057697563@qq.com，广州

🤖 Claude：[逐维确认教育背景、求职意向、项目经历...]
[Step 3] 结构设计：头部 → 教育背景 → 项目经历(60-70%) → 技能概要 → 竞赛证书
确认后进入生成。

🤖 Claude：[Step 4-7 重写项目描述 → 生成 DOCX → 验证 → 输出]
✅ 简历已生成：输出文件/杨子菁_简历.docx
```

## 文件结构

```
resume-expert/
├── SKILL.md     # Skill 定义（441 行，含全部逻辑、规则、矩阵、回退策略）
├── LICENSE      # MIT 协议
└── README.md    # 本文件
```

`SKILL.md` 就是整个 Skill——它是给 Claude 读的"操作手册"，没有额外代码。生成 docx 的代码由 Claude 根据 SKILL.md 中的规范实时编写。
