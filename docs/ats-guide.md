# ATS 兼容性深度指南

> 本文是 SKILL.md 中 ATS 章节的扩展阅读。如果你刚接触 ATS（Applicant Tracking System）概念，从这里开始。

## 什么是 ATS？

Applicant Tracking System——申请人追踪系统。大中型企业的招聘系统（如 Greenhouse、Lever、Workday、北森、Moka）用 ATS 自动解析和筛选简历。你的简历上传后，先被 ATS 解析成结构化数据，再按 JD 关键词匹配打分——**超过 75% 的简历在这一关被淘汰**。

## ATS 如何解析你的简历？

```
PDF/DOCX 上传
    ↓
文本提取（按阅读顺序，从左到右、从上到下）
    ↓
字段识别（姓名、电话、邮箱、教育、工作经历）
    ↓
关键词提取（技能名、工具名、方法论词汇）
    ↓
与 JD 匹配打分 → 排名 → HR 查看前 N 份
```

关键事实：**ATS 不理解排版美学**。它只看文本内容和结构。

## 常见 ATS 避坑清单

### 1. 表格布局 — 最致命的错误

```
❌ 双栏布局：
┌─────────────┬─────────────┐
│ 工作经历     │ 技能        │
│ ...         │ ...         │
└─────────────┴─────────────┘

ATS 按行读取：工作经历 | 技能 → 解析为"工作经历技能..."
破坏了所有逻辑分组。
```

### 2. 图片化信息

- ❌ 联系方式嵌在头像图片里 → ATS 完全看不到
- ❌ 技能水平用进度条/星级图表示 → 解析为空白
- ❌ 二维码 → ATS 输出乱码

### 3. 非标准字体

ATS 通常只支持系统默认字体回退。使用自定义字体可能导致：
- 特殊字符变成 □
- 中文字段识别失败
- 整个段落的文本丢失

### 4. 页眉/页脚

很多 ATS 解析器**直接忽略**页眉和页脚中的内容。如果把电话、邮箱放在页眉，ATS 可能认为你没有联系方式。

## 关键词优化进阶

### 动词 × 技术栈 组合密度

ATS 不仅看关键词是否出现，还看它们的**上下文质量**。散装关键词列表（如 `Java, MySQL, Redis, Docker`）的权重低于**叙事中的技术关键词**（如 "基于 Redis 实现分布式缓存，将 API 响应时间从 200ms 降至 15ms"）。

### JD 逆向工程

1. 找目标岗位的 JD，提取所有技术名词和方法论词汇
2. 按出现频率排序——出现 3 次以上的词是 **P0 关键词**，必须命中
3. 检查你的简历：每个 P0 关键词是否至少在项目描述中出现一次？如果是纯粹堆在技能区，ATS 给的权重会打折

### 常见 JD 关键词速查

| 岗位类型 | JD 高频词示例 |
|----------|-------------|
| 后端开发 | 分布式、高并发、微服务、Redis、消息队列、数据库优化 |
| 前端开发 | 组件化、性能优化、跨端、工程化、CI/CD |
| AI/算法 | 模型训练、特征工程、A/B 测试、推理优化、RAG |
| 产品经理 | 需求分析、数据驱动、PRD、用户调研、A/B 测试 |
| 设计 | 设计系统、组件库、用户研究、设计规范、Figma |

## ATS 测试

在实际投递前，可以这样测试：
1. 用纯文本工具打开你的 PDF/DOCX，看看内容是否完整、顺序是否合理
2. 复制全文到记事本——如果出现乱码、缺行、顺序混乱，ATS 也会遇到同样问题
3. 用各招聘平台的自带简历解析功能测试（如 BOSS 直聘的"上传简历"）——它们内部也是 ATS
