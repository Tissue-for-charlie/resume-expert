# 面试预测与 Q&A 生成完整规范

> 本文是 SKILL.md 模式 E 的扩展细则。包含简历信号提取、面试问题预测分类、Web 搜索增强策略、可交互 HTML Q&A 生成规范和与其他模式的联动规则。

## 设计动机

简历通过筛选只是求职第一步——面试官会针对简历内容做 3-4 层深度追问。许多候选人简历写得漂亮但面试翻车，根本原因是：

1. **简历内容超出自己的理解深度**：写了"精通 Kafka"但只跑过 demo，面试官一问"ISR 机制"就穿帮
2. **缺乏面试预期管理**：不知道面试官会针对哪些点追问，没有准备话术
3. **行业热点脱节**：简历技术栈和当前行业热点问题（如 2026 年的 LLM 落地、AI Agent 架构）脱节，面试官会问"你怎么看"

模式 E 的核心价值：**在简历投递前，预测面试官的高频追问，让候选人提前准备**。受 C1 不虚构原则约束——预测问题的目的是帮候选人准备真实经历的表达，不是教他编造答案。

## Step 1：简历信号提取

复用模式 B Step 1 已提取的简历文本，无需重复扫描。从简历中提取以下信号作为问题预测的输入：

### 1.1 信号类型

| 信号类型 | 提取内容 | 用于预测 |
|---------|---------|---------|
| **项目深挖信号** | 每个项目的核心技术、量化数据、角色描述 | 技术深度追问、架构决策追问 |
| **技能等级信号** | 技能清单中的等级词（精通/熟悉/了解） | 技能深度验证问题 |
| **时间线信号** | 工作经历时间区间、空窗期、跳槽频率 | 稳定性追问、空窗解释 |
| **量化数据信号** | QPS/DAU/GMV/转化率等数字 | 数据来源追问、方法论追问 |
| **职业轨迹信号** | 晋升路径、转行经历、行业切换 | 职业规划追问、转行动机 |
| **差异化信号** | 跨领域组合、独特项目、开源贡献 | 亮点深挖、个人故事追问 |

### 1.2 信号强度分级

每个信号按强度分级，强度越高越可能被追问：

| 强度 | 判定 | 处理 |
|------|------|------|
| 🔴 高强度 | 简历中明确写的核心技术/量化数据/等级词 | 必出预测问题，标注"高频追问" |
| 🟡 中强度 | 隐含但未展开的技术/经历 | 生成预测问题，标注"可能追问" |
| 🟢 低强度 | 简历未提但 JD 要求的 | 提示候选人"JD 要求但简历未写，面试可能被问" |

## Step 2：面试问题预测分类

预测问题按 6 大类组织，每类有独立的生成逻辑：

### 2.1 技术深度追问（基于项目）

针对简历中每个项目的核心技术栈，生成 3-4 层深度追问：

```
示例（简历写："基于 Spring Boot 完成电商交易模块，QPS 从 200 提升至 800"）：

L1 基础验证：「Spring Boot 在这个项目里主要用了哪些核心特性？」
L2 实现细节：「QPS 提升的具体手段是什么？是加缓存还是优化 SQL？」
L3 架构决策：「为什么选 Spring Boot 而不是 Spring Cloud？当时怎么权衡的？」
L4 极限场景：「如果 QPS 继续涨到 5000，你的架构还能撑住吗？瓶颈在哪？」
```

### 2.2 量化数据追问（基于数字）

针对简历中的每个量化数据，追问数据来源和计算方法：

```
示例（简历写："DAU 提升 30%"）：
- 「30% 的基准是什么？提升前后的绝对值是多少？」
- 「这个提升有多少归因于你的工作，多少是自然增长？」
- 「A/B 测试怎么做的？置信度多少？」
```

### 2.3 行为面试问题（基于职业轨迹）

针对跳槽、晋升、转行等轨迹，生成 STAR 框架行为问题：

```
示例（简历显示 2 年内跳槽 2 次）：
- 「请描述一次你和上级意见不一致的经历，你怎么处理的？」
- 「为什么频繁换工作？每次离职的核心原因是什么？」
- 「如果这份工作遇到类似情况，你会怎么应对？」
```

### 2.4 行业认知问题（基于行业热点）

基于行业矩阵和当前热点，生成行业趋势问题：

```
示例（互联网后端岗）：
- 「你怎么看 Service Mesh 在中小公司的落地价值？」
- 「LLM 对后端架构的影响，你的判断是什么？」
- 「微服务拆分的边界，你实践中怎么把握？」
```

### 2.5 弱势因素追问（基于弱势群体策略）

如检测到空窗期/双非/大龄等弱势因素，生成针对性追问及应对建议：

```
示例（检测到 6 个月空窗期）：
- 「2024.03-2024.09 这段时间在做什么？」
- 「这段时间有没有持续学习？学了什么？」
- 「为什么没有立即找工作？」

每题附带「建议回答方向」（来自 low-info-strategies.md 的面试话术模板）
```

### 2.6 反向提问准备

面试末尾面试官常问"你有什么想问的"——基于简历内容生成建议的反向提问：

```
- 基于候选人关注的职业发展：「贵团队的技术栈演进方向是什么？我有机会参与架构升级吗？」
- 基于项目深度：「贵公司在 [候选人项目相关领域] 的下一步规划是什么？」
- 基于弱势因素（谨慎使用）：「团队的平均工作年限大概是多少？」（间接了解稳定性文化）
```

## Step 3：Web 搜索增强（如可用）

WebSearch 工具可用时，执行以下搜索以增强预测准确性：

### 3.1 搜索策略

| 搜索目的 | Query 模板 | 用途 |
|---------|-----------|------|
| 目标公司面经 | `"{公司名} {岗位} 面经 {当前年份}"` | 获取真实面试题库 |
| 岗位通用面试题 | `"{岗位} 面试题 高频 {当前年份}"` | 补充通用问题 |
| 技术热点 | `"{核心技术} 面试深度问题 {当前年份}"` | 更新技术深度问题 |
| 行业趋势 | `"{行业} 技术趋势 {当前年份}"` | 更新行业认知问题 |
| JD 对标 | `"{公司名} {岗位} JD 要求 {当前年份}"` | 补充 JD 隐性要求 |

### 3.2 搜索结果处理

- **去重**：相同问题合并，保留最高频表述
- **分类**：按 2.1-2.6 分类体系归类
- **时效性**：优先近 1 年的面试题，技术热点按当前年份更新
- **可信度**：多来源验证的问题标记"高频"，单一来源的标记"参考"

### 3.3 不可用时回退

WebSearch 不可用时（离线 / 非 Claude Code 平台），完全依赖：
- Step 2 的简历信号提取 + 分类生成逻辑
- 内置的行业通用面试题库（见下文）

### 3.4 内置通用面试题库（回退用）

| 岗位类型 | 高频通用问题 |
|---------|------------|
| 后端开发 | 「CAP 理论」「分布式锁实现」「数据库索引优化」「消息队列对比」 |
| 前端开发 | 「浏览器渲染流程」「React/Vue diff 算法」「性能优化手段」 |
| 产品经理 | 「需求优先级怎么排」「数据驱动决策案例」「如何处理需求冲突」 |
| 数据分析 | 「SQL 窗口函数」「AB 测试设计」「指标体系搭建」 |
| 设计师 | 「设计系统构建」「用户研究方法」「设计决策权衡」 |

## Step 4：可交互 HTML Q&A 生成

将预测问题生成为**单个自包含 HTML 文件**，用户可在浏览器打开交互练习。

### 4.1 HTML 设计规范

**核心原则**：
- 单文件自包含——所有 CSS/JS 内联，无外部依赖，可离线使用
- 移动端适配——支持手机浏览（求职者常在通勤时复习）
- 交互式——折叠/展开答案、标记掌握程度、搜索过滤
- 视觉清晰——按问题类型分色、按强度标记优先级

### 4.2 页面结构

```
┌─────────────────────────────────────────┐
│  🎯 面试预测 Q&A — {岗位} @ {公司}        │
│  📊 共 N 题 | 🔴 高频 X | 🟡 中频 Y | 🟢 低频 Z │
│  [搜索框] [全部展开] [全部折叠] [导出]      │
├─────────────────────────────────────────┤
│  📁 分类筛选：[技术深度] [量化数据] [行为]   │
│              [行业认知] [弱势因素] [反向提问] │
├─────────────────────────────────────────┤
│  ┌───────────────────────────────────┐  │
│  │ 🔴 [技术深度] L2 实现细节              │  │
│  │ Q: QPS 提升的具体手段是什么？          │  │
│  │ [展开答案 ▼] [✓ 已掌握] [✗ 需复习]   │  │
│  └───────────────────────────────────┘  │
│  ┌───────────────────────────────────┐  │
│  │ 🟡 [行为面试] 稳定性追问              │  │
│  │ Q: 为什么频繁换工作？                │  │
│  │ [展开答案 ▼] [✓ 已掌握] [✗ 需复习]   │  │
│  └───────────────────────────────────┘  │
├─────────────────────────────────────────┤
│  📊 学习进度：已掌握 X/N | 需复习 Y/N       │
└─────────────────────────────────────────┘
```

### 4.3 答案区设计

每个问题的答案区包含三部分：

```
┌─────────────────────────────────────────┐
│ 💡 建议回答方向（不是标准答案）            │
│ ─────────────────────────────            │
│ 回答要点：                                │
│ 1. 先给出结论性回答（30 秒内）              │
│ 2. 用简历中的项目作为论据                  │
│ 3. 量化数据补充细节                        │
│                                          │
│ 📝 你的简历相关内容：                      │
│ [引用简历原文]                             │
│                                          │
│ ⚠️ 避免陷阱：                              │
│ - 不要说"这个我没深入了解"——会被认为不匹配  │
│ - 不要编造数据——面试官会追问数据来源        │
│                                          │
│ 🔗 延伸学习（如有 Web 搜索结果）：           │
│ [相关面经链接]                             │
└─────────────────────────────────────────┘
```

### 4.4 交互功能

| 功能 | 实现 | 用途 |
|------|------|------|
| **折叠/展开** | 点击问题卡片切换答案区可见性 | 分段学习 |
| **掌握度标记** | 三档按钮（已掌握/需复习/未学习） | 进度追踪 |
| **搜索过滤** | 输入关键词实时过滤问题 | 定位复习 |
| **分类筛选** | 点击分类标签筛选问题类型 | 分类专练 |
| **强度筛选** | 按红/黄/绿强度筛选 | 优先复习高频题 |
| **进度统计** | 底部固定栏显示掌握度 | 学习激励 |
| **本地存储** | localStorage 保存掌握度标记 | 跨会话保留 |
| **导出** | 一键导出为 Markdown/打印 | 离线复习 |

### 4.5 视觉规范

- **配色**：与简历 docx 配色一致（#333333/#1A1A1A/#666666/#2B579A）
- **字体**：系统默认（微软雅黑/Arial），无需加载外部字体
- **问题强度配色**：🔴 #E74C3C / 🟡 #F39C12 / 🟢 #27AE60
- **分类标签配色**：6 类各一色，浅色背景 + 深色文字
- **响应式**：移动端单列、桌面端可双列

### 4.6 生成与保存

- 生成文件名：`面试预测_{岗位}_{公司}_{日期}.html`
- 默认保存到 `outputs/` 目录
- 生成后自动在浏览器打开（如环境支持）
- 非 Claude Code 平台：输出 HTML 源码让用户自行保存

## Step 6：与其他模式联动

### 6.1 模式 B → 模式 E

模式 B 评审完成后，主动询问是否生成面试预测：
- 复用模式 B 已提取的简历文本，无需重复扫描
- 模式 B 的"总结亮点"直接作为模式 E 的差异化信号输入
- 模式 B 的"必须改"项 → 模式 E 的"高风险预警"项

### 6.2 模式 D → 模式 E

模式 D 完成 JD 匹配度扫描后，主动询问是否生成面试预测：
- 基于 JD 的硬性要求 + 加分项，补充 JD 相关面试题
- 模式 D 的红旗项 → 模式 E 的高风险追问项
- 模式 D 的竞争水位估计 → 模式 E 的面试难度预期

### 6.3 模式 A → 模式 E

模式 A 生成简历后，主动询问是否生成面试预测：
- 帮助用户在投递前评估"这份简历会不会在面试翻车"
- 如发现简历内容超出用户理解深度（如 Skill 自动补充的技术词），高亮预警

### 6.4 模式 E → 模式 C

模式 E 完成后，如发现简历内容有"面试高风险"项（如写了"精通"但实际不熟），主动建议：
- 「简历中"精通 Kafka"可能引发深度追问，建议改为"熟悉"或补充项目深度」
- 「空窗期 6 个月建议在简历中补充说明，避免面试被动应对」
- 切换到模式 C 修改简历内容

### 6.5 弱势群体策略联动

模式 E 自动检测简历中的弱势因素（空窗期/双非/大龄），生成针对性追问并附带应对建议：
- 空窗期追问 → 引用 [`low-info-strategies.md`](low-info-strategies.md) 的面试话术模板
- 双非背景追问 → 引用双非面试应对预案
- 大龄求职者追问 → 引用大龄面试应对预案

## Step 7：约束与伦理

### 7.1 C1 不虚构原则延伸

模式 E 的答案区是「建议回答方向」而非「标准答案」：
- ❌ 禁止：生成完整的"标准答案"让候选人背诵——这会导致面试回答千篇一律且与候选人真实经历脱节
- ✅ 正确：给出回答框架（结论 → 论据 → 数据），让候选人基于自己的真实经历组织答案

### 7.2 不诱导编造

- Web 搜索到的面经题，答案区标注"此题来自面经，请结合你的真实经历回答"
- 如候选人简历中无相关经历，标注"此题超出你当前简历范围，建议面试时坦诚说明并表达学习意愿"
- 不生成"如何编造经历应对此问题"的指导

### 7.3 心理健康提示

面试准备可能引发焦虑。HTML 文件底部附加提示：

```
💡 复习建议：
- 不要试图背答案——面试官能识别模板化回答
- 重点是理解回答框架，用自己的话表达
- 高频题优先复习，但不要忽视中低频题
- 面试前一晚保证睡眠，比多看一道题更重要
- 如果某类问题让你特别焦虑，可能是简历该调整的信号——考虑模式 C 优化简历
```

```html
          <div class="actions">
            <button onclick="mark(this,'mastered')">✓ 已掌握</button>
            <button onclick="mark(this,'review')">✗ 需复习</button>
          </div>
        </div>
      </div>
    </div>
  </main>
  <footer class="progress-bar">
    <div>已掌握 X/N | 需复习 Y/N</div>
    <div class="tips">💡 复习建议...</div>
  </footer>
  <script>
    /* 内联 JS：折叠展开、标记、搜索过滤、localStorage */
  </script>
</body>
</html>
```

**关键 JS 功能**：
- `toggleAnswer(el)`：切换答案区可见性
- `mark(el, status)`：标记掌握度并存 localStorage
- `filterQuestions(category, intensity)`：按分类/强度筛选
- `searchQuestions(keyword)`：实时搜索过滤
- `updateProgress()`：更新底部进度统计
- `exportMarkdown()`：导出为 Markdown
