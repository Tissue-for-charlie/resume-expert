# 极简用户输入应对策略

> 本文是 SKILL.md「极简用户输入应对策略」和「低年级/转行候选人特殊策略」的完整扩展。

## 触发条件

当 Step 1 压缩信息块解析后（或正常收集后）全维度已填 ≤4 个时（如只有「姓名+电话+学校+岗位」），启用以下策略。

## 信息收集阶段的调整

- **精简提问维度**：跳过「竞赛/证书」「到岗时间」「核心课程」等低优先级维度——直接不提问
- **一次问完剩余所有维度**：不逐维度推进，改为一次性抛出剩余问题的合并提问
- **接受"没有"**：用户连续两个维度说"没有"后，跳过剩余非必要维度，直接进入结构设计

## 项目经历的深度展开

只有一个项目时（如"给B站UP主做过50多张封面"），展开策略：

```
原始信息：给B站UP主做过超50张视频封面，深度使用ChatGPT和Gemini，以及即梦，生产效率高

展开方向：
1. 工作流 → 需求沟通 → AI创意构思 → AI生成素材基底 → 手动精细排版 → 交付迭代
2. 工具链 → ChatGPT/Gemini（创意方向） + 即梦（AI文生图） + PS（合成调色光影） + AI（字效版式）
3. 量化 → 50+ 张累计交付、覆盖 N 个内容赛道、持续合作复购
4. 风格 → 知识科普 / 生活Vlog / 游戏实况 / 其他——每条赛道审美逻辑不同
5. 效率 → AI工具链缩短单张产出周期
6. 软技能 → 远程协作、理解UP主需求、自主管理排期
```

核心原则：
- 一条原始描述 → 4-6 条 bullet，每条从不同切面切入
- 不要虚构不存在的信息，但把用户已提及的关键词（工具名、数量、效率描述）充分展开
- 禁止把同一条信息拆成两句充数——每条 bullet 必须是独立的信息点

## 技能概要区的补偿

用户信息少时，技能概要区的作用从"补充验证"升级为"信息聚合"：

- 从用户的每一句描述中提取技能关键词汇入技能概要（如"即梦"→AI 文生图；"生产效率高"→高效设计工作流）
- 技能概要分组用 `**标签**: 子项` 的格式，让少量技能词在视觉上显得有结构
- 可以加一行「核心能力」把软技能（风格适配、AI辅助创意）聚合呈现

## 排版补偿

用户信息少 → 一页 A4 可能填不满。补偿方式：

- 头部信息行用更大的字号和间距撑起顶部视觉重量
- 项目经历 bullet 间距稍大（50-60 twips），单段内容深但呼吸感好
- 不强行添加无意义的占位段落——留白比废话好

## 低年级/转行候选人的特殊策略

当用户是低年级在校生或从其他行业转行时，天然面临"经验少"的劣势。应对策略：

- **用项目经历碾压**：一个完整架构、有量化成果、持续维护的项目，比绩点更能证明能力
- **不要隐藏 AI 或工具的辅助**：不写成"我用了 AI 写代码"，而是"N 轮迭代驱动的开发方法论"——方法论比单次结果更能体现工程思维
- **竞赛和证书靠后放**：它们是佐证，不是核心卖点
- **诚实但有策略**：不虚报经验，但可以强调"持续维护的项目"+"跨领域技能组合"——这些都是可靠的正面信号

## 单项目简历策略

- 允许扩展到 5-6 条 bullet，从不同切面展开：**工作流与方法论** → **工具深度** → **量化成果** → **风格与广度** → **软技能信号**
- 禁止把同一条信息拆成两句充数
- 单项目简历的技能概要区需要略长

## 兼职/自由职业的项目归类

- 重点在"做了多少张/用的什么工具/什么风格"→ 归为**项目经历**
- 重点在"公司名/岗位/时间段/月薪"→ 归为**工作/实习经历**
- 兼职场景默认优先用「项目经历」承载——兼职本质上是一系列零散项目的集合
- 如果用户同时有正式实习和兼职项目，正式实习放"工作/实习经历"，兼职放"项目经历"
