# 评分系统完整规范

> 本文是 SKILL.md 模式 B 的评分细则扩展。包含评分维度、行业权重映射、分数段锚点、基准对比和反馈输出格式。

## 评分维度与权重（默认）

| 维度 | 权重 | 考察点 |
|------|------|--------|
| 内容质量 | 40 | 项目描述是否有结构和量化数据；关键信息是否完整（联系方式/GitHub/作品链接） |
| 排版可读性 | 25 | 字号层级是否清晰；是否存在 info dump（大段连续文字无换行）；页面空间利用是否高效；**页数是否超标** |
| 关键词覆盖 | 20 | 目标行业的核心关键词是否命中（技术岗：框架/协议/工具链；产品岗：数据指标/方法论；金融岗：证书/分析工具） |
| 差异化定位 | 15 | 面试官能否找到"想深入聊聊"的钩子（跨领域组合、独特的项目经历、同龄人不具备的能力） |

### 五维评分（娱乐/演艺/媒体行业专用）

对于娱乐/演艺/媒体行业，标准四维评分框架无法充分衡量"传播影响力"——这是该行业最核心的能力信号。因此为这些行业引入五维评分：

| 维度 | 权重 | 考察点 |
|------|------|--------|
| 内容质量 | 30 | 个人简介是否有定位；演出/项目描述是否有结构和传播数据；关键信息是否完整 |
| 排版可读性 | 20 | 字号层级是否清晰；影响力数据/奖项/作品模块是否视觉突出；**页数是否超标** |
| 影响力数据 | 20 | 播放量、粉丝数、榜单排名、热搜次数、演唱会规模等传播指标；数据是否具体（"全网 20 亿+"优于"数据亮眼"） |
| 关键词覆盖 | 10 | 目标领域的核心关键词是否命中（曲风/平台/综艺/奖项类型） |
| 差异化定位 | 20 | 面试官/制作人能否找到"想了解一下"的钩子——独特嗓音标签（空灵/戏剧性/多变）、跨曲风跨度、多语言能力、[科班背景+流行出圈]的组合 |

**评分说明**：五维评分满分仍为 100 分，分数段锚点与四维评分共用。适用于娱乐/演艺、媒体/新闻及相近行业。

## 分数段锚点

保证不同对话中评分一致：

| 分数段 | 含义 | 典型特征 |
|--------|------|----------|
| 90–100 | 行业标杆 | 4 个维度均无明显短板；项目描述有结构、有数据、有叙事；版面高效且关键词覆盖到位；有明确差异化钩子 |
| 75–89 | 良好，可优化 | 1–2 个维度存在中等短板；有优化方向但不需要推倒重来 |
| 60–74 | 及格线附近 | 多项维度存在明显问题（info dump、关键信息缺失、无量化数据）；建议结构级调整 |
| 40–59 | 多处硬伤 | 至少 2 项维度需要重建；大幅重写优于局部修补 |
| < 40 | 建议从零重写 | 缺失联系方式/GitHub 等关键信息，或内容几乎无法支撑目标行业的筛选门槛 |

## 同岗位基准对比

评分完成后，在评分明细表下方追加一行基准参考，帮助用户理解在竞争中的相对位置：

```
📊 同岗位对比：您的得分 65 | 同岗位中位数 58 | 前 25% 门槛 68
→ 您处于中上水平（高于中位数 7 分），距前 25% 还差 3 分。建议聚焦关键词覆盖和差异化定位两个维度。
```

## 分级反馈

- 🔴 **必须改**：事实错误、关键信息缺失（联系方式/GitHub/作品集）、排版硬伤（字体 null、页边距异常）、**页数超标**（0–5 年经验超过一页，或 5–15 年经验超过两页）
- 🟡 **建议改**：描述冗长、缺量化数据、竞赛名称占用过多空间、HR 大概率跳读的长段落、**页面临界溢出**

### 页数评估

评分表的评分明细下方，输出页数评估：

```
📄 页数评估：1 页 ✓（约 1500 字，排版紧凑可控）
📄 页数评估：1 页 ⚠️（约 1950 字，目前可排下但内容偏满，删减任一项目后会更舒适）
📄 页数评估：溢出至第 2 页 🔴（约 2200 字，必须精简。建议：① 精简项目描述 ② 缩减次要项目 bullet ③ 合并证书行）
```

## 反馈输出格式（四列模板）

每条反馈必须填满四列：

```
| 原文 | 问题 | 建议替换 | 替换理由 |
|------|------|----------|----------|
| "负责XX模块的开发和维护工作" | 被动罗列，无技术细节 | "基于 Spring Boot 独立完成 XX 模块，设计 RESTful API 12 个，QPS 从 200 提升至 800" | 主动动词 + 技术栈 + 量化产出，面试官可追问架构决策 |
```

如果某列确实无内容（如原文缺失该信息），写 `—`，不空置。

## 评审后总结

列出 3-5 条面试官会产生"想聊一下"冲动的信号（仅当确实存在）。如果用户经历平庸，不强行夸——直接告诉用户哪些方向值得后续强化。

## 评分前校准（AI 自检流程）

在每次评审用户简历之前，请执行以下校准步骤以保证评分一致性：

1. **找锚点**：从 [`scoring-golden-tests.md`](scoring-golden-tests.md) 中找到与目标行业/职级最接近的 2-3 个 Golden Test 用例（同行业优先，相邻行业其次），速读其评分明细——建立"这个分数段的简历长什么样"的基准感
2. **对齐权重**：确认目标行业的四维权重映射（见上文行业 × 评分权重映射表），不要用默认权重（互联网）去评特殊行业（如金融、设计）
3. **评分后自检**：打完分后，将用户简历与同分数段的 Golden Test 简历做一次对比——它们的质量水平是否确实相近？
4. **允许 ±5 波动**：评分存在主观成分，同一份简历在不同对话中 ±5 分属于可接受范围。如果波动超过 5 分，说明评分标准发生了漂移，应重新校准
5. **🌐 Web 搜索增强（可选）**：当 WebSearch 工具可用时，建议在评分前搜索目标行业的当前市场数据：
   - 搜索 query 1：`"{行业} {职级} 简历 核心关键词 {当前年份}"`
   - 搜索 query 2：`"{行业} 招聘 薪资 竞争 报告 {当前年份}"`
   - 搜索结果用途：热门技术栈/证书/量化指标 → 校准**关键词覆盖**维度；薪资与竞争报告 → 校准**基准中位数**和**差异化定位**维度
   - **不可用时回退**：WebSearch 不可用时（如离线/非 Claude Code 平台），完全依赖 `scoring-golden-tests.md` 中的 75 份静态 Golden Test 进行校准

> 📖 [`scoring-golden-tests.md`](scoring-golden-tests.md) — 75 份金标准简历 + 专家评分明细，覆盖 15 个行业 × 5 个分数段（90-100/75-89/60-74/40-59/<40），含 Web 搜索增强校准指引

## 与模式 D（JD 匹配度扫描）的边界

模式 B 和模式 D 评分目标不同，**不可互相替代**：

| 维度 | 模式 B（行业基准评分） | 模式 D（JD 匹配度扫描） |
|------|------------------------|--------------------------|
| 评估标尺 | 对标行业基准（业内"应当达到什么水平"） | 对标具体 JD（这份简历投这个岗位的命中率） |
| 输入要求 | 简历 + 目标行业 + 职级 | 简历 + 目标 JD 文本/链接 |
| 评分维度 | 内容质量 / 排版 / 关键词 / 差异化 | 硬性门槛 / 关键词覆盖 / 经验匹配 / 资质匹配 / 加分项覆盖 |
| 评分基准 | Golden Test 75 份校准简历 | JD 中提取的硬性 + 加分项清单 |
| 输出形式 | 100 分总分 + 同岗位基准对比 + 分级反馈 | 100 分匹配度 + 红旗清单 + 关键词命中表 + 优化建议 |
| HR 视角 | 候选人在竞争池中的相对位置 | 候选人针对具体岗位能否通过筛选 |

**模式 B → D 衔接规则**：模式 B 完成后若用户提供目标 JD，**不重复评分**，而是切换到模式 D 的五维匹配度算法，并在报告中注明"基于模式 B 已提取的简历文本"。

**模式 D 不可替代模式 B**：JD 匹配度无法回答"我的简历在竞争池中处于什么位置"——这是模式 B 的核心价值。建议流程：先模式 B 看行业水平 → 再模式 D 看岗位命中率 → 视情况进入模式 C 优化。

详见 [`jd-matching.md`](jd-matching.md) 完整规范。
